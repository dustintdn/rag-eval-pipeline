# Evaluating RAG Pipelines

## Why Evaluation Matters

A RAG system can fail silently. The retriever might fetch irrelevant chunks. The LLM might generate confident-sounding but unfaithful answers. Without systematic evaluation, these failures are invisible until a user reports them.

Evaluation closes the feedback loop: it lets you measure the impact of changes to chunk size, top-k, prompt templates, and model choice before deploying them.

## Retrieval Metrics

### Hit Rate
The fraction of questions for which at least one retrieved chunk contains the ground truth answer. A hit rate of 0.8 means 80% of questions had relevant context available to the LLM.

**Formula:** hits / total questions

### Mean Reciprocal Rank (MRR)
For each question, MRR measures the rank of the first relevant chunk. A chunk at rank 1 contributes 1.0; rank 2 contributes 0.5; rank 3 contributes 0.33. MRR penalizes systems that bury the relevant chunk.

**Formula:** mean(1 / rank_of_first_relevant_chunk)

### NDCG (Normalized Discounted Cumulative Gain)
A graded ranking metric that rewards systems that place the most relevant chunks at the top. Unlike MRR, it accounts for multiple relevant chunks.

## Generation Metrics (RAGAS)

RAGAS is an open-source framework for evaluating RAG pipelines using LLM-as-judge techniques.

### Faithfulness
Measures whether every claim in the generated answer is grounded in the retrieved context. Score of 1.0 means every statement is directly supported. Score below 0.7 typically indicates hallucination.

### Answer Relevancy
Measures whether the answer addresses the question. A high score means the answer is on-topic; a low score means the model went off on a tangent or answered a different question.

### Context Precision
Of the retrieved chunks, what fraction were actually relevant? High precision means the retriever isn't flooding the context with noise.

### Context Recall
Does the retrieved context contain all the information needed to answer the question? Low recall means the retriever missed key chunks.

## Eval Dataset Construction

A good eval dataset is:
- **Diverse** — covers different question types (factual, comparative, multi-hop)
- **Challenging** — includes questions that require synthesizing multiple chunks
- **Realistic** — questions that actual users would ask

For each sample you need: question, ground truth answer, the retrieved context chunks, and the model's generated answer. The ground truth is hand-written; the rest can be generated by running the pipeline.

## Interpreting Results

| Score | Interpretation |
|---|---|
| > 0.85 | Production-ready |
| 0.70 – 0.85 | Good, iterate on weakest metric |
| 0.50 – 0.70 | Needs significant improvement |
| < 0.50 | Fundamental pipeline issue |

## Common Failure Modes

- **Low faithfulness + high relevancy**: Model answers well but hallucinates. Fix: tighten the system prompt to restrict the model to only what's in context.
- **High faithfulness + low recall**: Model stays grounded but retriever is missing key chunks. Fix: increase top-k or reduce chunk size.
- **Low context precision**: Retriever returns noise. Fix: use a reranker or raise the similarity threshold.
